掌握核心技术之Redis

2021年1月13日
| 中间件

Redis 是一个开源（BSD许可）的，内存中的数据结构存储系统，它可以用作数据库、缓存和消息中间件。

SCAN命令

用于迭代数据库中的数据库键，替换keys命令

  • SSCAN 命令用于迭代集合键中的元素。
  • HSCAN 命令用于迭代哈希键中的键值对。
  • ZSCAN 命令用于迭代有序集合中的元素（包括元素成员和元素分值）。

GEO

主要用于存储地理位置信息，并对存储的信息进行操作

Bitmaps位图

日活跃用户统计：

redis.setbit(daily_active_users, user_id, 1) # 添加记录
redis.bitcount(daily_active_users) # 统计记录

应用场景：

  • 存储用户的签到情况
  • 文章已读或未读情况
  • 统计今日登录的用户数

HyperLogLog

HyperLogLog其实是一种算法，它是用来估算一个集合的基数的。

应用场景：

  • 统计页面访问的UV数
  • 统计网站的ip数

布隆过滤器

原本有10亿个号码，现在又来了10万个号码，要快速准确判断这10万个号码是否在10亿个号码库中。

  • bf.add 添加元素到布隆过滤器
  • bf.exists 判断某个元素是否在布隆过滤器中

布隆过滤器说某个元素在，可能会被误判。布隆过滤器说某个元素不在，那么一定不在。

延时队列

通过有序集合(sorted set)实现

  • zadd key score1 value1 赋值
  • zrevrange key start end withscores 查询成员分数从大到小顺序的索引 start 到 end 的所有成员。
  • zrem key value1 value2 删除指定元素
type delayFunc func(message interface{})

type Process struct {
	queue       string
	job         delayFunc
	concurrency int
}

func NewConsumer(queue string, delayFunc delayFunc, concurrency int) *Process {
	return &Process{
		queue:       queue,
		job:         delayFunc,
		concurrency: concurrency,
	}
}

func NewProducer(queue string) *Process {
	return &Process{
		queue: queue,
	}
}

func (process *Process) getTask() ([]redis2.Z, error) {
	opt := redis2.ZRangeBy{
		Min:    "0",
		Max:    strconv.FormatInt(time.Now().Local().UnixNano(), 10),
		Offset: 0,
		Count:  1,
	}
	return redis.Cli.ZRangeByScoreWithScores(process.queue, opt).Result()
}

func (process *Process) delTask(member interface{}) (int64, error) {
	return redis.Cli.ZRem(process.queue, member).Result()
}

func (process *Process) AddTask(members redis2.Z) error {
	return redis.Cli.ZAdd(process.queue, members).Err()
}

func (process *Process) ConsumerTask() {
	for {
		tasks, err := process.getTask()
		if err != nil {
			logrus.Errorf("ConsumerTask GetTask err: %s", err.Error())
		}

		if len(tasks) <= 0 {
			time.Sleep(time.Second * 1)
			continue
		}

		del, err := process.delTask(tasks[0].Member)
		if err != nil {
			logrus.Errorf("ConsumerTask DelTask err: %s, Member: %s", err.Error(), tasks[0].Member)
			time.Sleep(time.Second * 1)
			continue
		}

		if del > 0 {
			logrus.Infof("【%d】ConsumerTask ReceiveMessage: %v", common.GetGoroutineID(), tasks[0].Member)
			process.job(tasks[0].Member)
		}
	}
}

func (process *Process) Run() {
	for i := 0; i < process.concurrency; i++ {
		go process.ConsumerTask()
	}
	select {}
}

分布式锁

分布式锁一般有三种实现方式：1. 数据库乐观锁；2. 基于Redis的分布式锁；3. 基于ZooKeeper的分布式锁。

数据库乐观锁

// step1: 查询出商品信息
select (quantity,version) from items where id = 100;
// step2: 根据商品信息生成订单
insert into orders(id, item_id) values(null, 100);
// step3: 修改商品的库存
update items set quantity = quantity - 1, version = version + 1 where id = 100 and version = #{version};

如果每次访问冲突概率小于20%，推荐使用乐观锁，否则使用悲观锁。乐观锁的重试次数不得小于3次。

基于Redis的分布式锁

  • 加锁：判断 key 是否存在、保存 key-value、设置 key 的过期时间
  • 解锁：判断 key 是否存在、判断是否自己持有锁、删除 key-value

注意事项：

  • 如果key存在，但是没有剩余时间。
  • 加锁的时候的 value 必须是一个全局唯一的字符串。
  • 防止删除重新加锁的 key
type Rds struct {
	Client         *redis.Client
	AcquireTimeout int32
	LockTimeout    int32
}

func NewRds(host, passwd string, db int) (rds *Rds, err error) {
	rdsOpt := &redis.Options{
		Addr:     host,
		Password: passwd,
		DB:       db,
	}
	client := redis.NewClient(rdsOpt)
	_, err = client.Ping().Result()
	if err != nil {
		return
	}
	rds = &Rds{Client: client, AcquireTimeout: 10, LockTimeout: 10}
	return
}

func (r *Rds) Lock(key string) (identifier string, b bool) {
	identifier = uuid.NewV4().String()
	lockName := "lock:" + key
	end := time.Now().Add(time.Second * time.Duration(r.AcquireTimeout))
	for time.Now().Before(end) {
		if r.Client.SetNX(lockName, identifier, time.Second*time.Duration(r.LockTimeout)).Val() {
			// 如果key不存在，并成功设置了key
			b = true
			return
		} else if r.Client.TTL(lockName).Val() == -1 {
			// 如果key存在，但是没有剩余时间
			r.Client.Expire(lockName, time.Second*time.Duration(r.LockTimeout))
		}
		time.Sleep(time.Microsecond)
	}
	return
}

func (r *Rds) UnLock(key, identifier string) (b bool) {
	lockName := "lock:" + key
	txf := func(tx *redis.Tx) error {
		v, err := tx.Get(lockName).Result()
		if err != nil {
			return err
		}

		_, err = tx.Pipelined(func(pipe redis.Pipeliner) error {
			if v == identifier {
				pipe.Del(lockName)
				b = true
			}
			return nil
		})

		return err
	}
	for {
		err := r.Client.Watch(txf, lockName) // 在事务执行之前被其他命令所改动，那么事务将被打断
		if err == nil { // 解锁成功，跳出循环
			break
		} else if err == redis.TxFailedErr {
			log.Fatal(err)
		}
	}
	return
}

另外一个版本：

var rdb *redis.Client
var ctx = context.Background()
var mutex sync.Mutex

func Init() {
	rdb = redis.NewClient(&redis.Options{
		Addr:     "localhost:6379",
		Password: "",
		DB:       0,
	})
}

func Lock(key string) bool {
	mutex.Lock()
	defer mutex.Unlock()
	result, err := rdb.SetNX(ctx, key, 1, 10*time.Second).Result()
	if err != nil {
		logrus.Error(err)
	}
	return result
}

func UnLock(key string) int64 {
	nums, err := rdb.Del(ctx, key).Result()
	if err != nil {
		logrus.Error(err)
		return 0
	}
	return nums
}

基于ZooKeeper的分布式锁

ZooKeeper分布式锁是基于临时顺序节点来实现的

  • 临时顺序节点自动释放锁
  • 监听前一个节点的变化
func main() {
	c, _, err := zk.Connect([]string{"127.0.0.1"}, time.Second) //*10)
	if err != nil {
		panic(err)
	}
	l := zk.NewLock(c, "/lock", zk.WorldACL(zk.PermAll))
	err = l.Lock()
	if err != nil {
		panic(err)
	}
	println("lock success, do your business logic")

	time.Sleep(time.Second * 10)

	l.Unlock()
	println("unlock success, finish business logic")
}

多线程

Redis支持多线程主要就是两个原因：

  • 可以充分利用服务器CPU资源，目前主线程只能利用一个核。
  • 多线程任务可以分摊Redis同步IO读写负荷。

Redis的多线程部分只是用来处理网络数据的读写和协议解析，执行命令仍然是单线程顺序执行。

IO多路复用： 多路指的是多个Socket连接，复用指的是复用一个线程。多路复用主要有三种技术：Select，Poll，Epoll，Epoll是最新的也是目前最好的多路复用技术。

采用多路I/O复用技术和内存操作造就Redis具有很高的吞吐量。

单线程的Redis为什么这么快

  • 纯内存操作
  • 单线程操作，避免了频繁的上下文切换
  • 采用了非阻塞I/O多路复用机制

持久化

快照形式RDB和增量文件AOF。AOF是一种追加执行命令的形式，它跟RDB的区别是，AOF并不是把数据保存下来，而是保存执行的动作。

如果appendonly.aof和dump.rdb文件同时存在，Redis是以AOF为优先的，毕竟AOF相对RDB来说在出现异常的情况下保存的数据更加完整。

快照RDB

自动重写rdb快照文件

save 900 1
save 300 10
save 60 10000

多少时间以内多少个key发生变化就会触发快照

手动重写rdb快照文件

./redis-cli -a pwd bgsave

save命令会堵塞主进程，bgsave生成一个子进程创建rdb文件。

增量文件AOF

自动重写aof文件

aof的触发条件配置项，下面配置三选一

appendfsync always   # 每次接收到命令都写入磁盘，保证数据安全性，但比较消耗服务器资源
appendfsync everysec # 每秒钟向磁盘中写入一次，推荐使用此配置
appendfsync no       # 完全依赖os，但性能最好

手动重写aof文件

./redis-cli -a pwd bgrewriteaof

集群方式

redis有三种集群方式：主从复制，哨兵模式和集群模式。

主从复制

当从数据库启动时，会向主数据库发送sync命令，主数据库接收到sync后开始在后台保存快照rdb，在保存快照期间收到的命令缓存起来，当快照完成时，主数据库会将快照和缓存的命令一块发送给从数据库。复制初始化结束。之后，主每收到1个命令就同步发送给从。

哨兵模式

在主从复制的基础上，哨兵实现自动化的故障恢复。

配置哨兵监控一个系统时，只需要配置其监控主数据库即可，哨兵会自动发现所有复制该主数据库的从数据库。

集群模式

插槽（slot）的取值范围是：0-16383。当我们的存取的key到达的时候，redis会根据crc16的算法得出一个结果，然后把结果对 16384 求余数，这样每个 key 都会对应一个编号在 0-16383 之间的哈希槽，通过这个值，去找到对应的插槽所对应的节点，然后直接自动跳转到这个对应的节点上进行存取操作。
